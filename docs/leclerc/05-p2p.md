# 05 · P2P — Delegated Inference & Dead-Drop Messaging

> Two P2P features, both over the Holepunch/Hyperswarm DHT that QVAC already bundles. **Spike this first (day 1–2)** — it's the riskiest novel piece. See [10](./10-build-plan.md).

## 1. Delegated inference (mobile → station)

QVAC has this built in. The station advertises itself as a provider; the mobile/thin client delegates heavy `completion`/`embed`/`rag*`/`ocr` to it.

### Station (provider) — server-only, runs on the laptop
```ts
import { startQVACProvider, stopQVACProvider, heartbeat } from "@qvac/sdk";

const { success, publicKey } = await startQVACProvider({ /* ProvideParams */ });
// publicKey is the station's peer id. Identity is controlled by QVAC_HYPERSWARM_SEED (64-hex).
// Print/QR this publicKey for pairing. Persist the seed so the key is stable across restarts.
```
- Set `QVAC_HYPERSWARM_SEED` so the station's `publicKey` is **stable** (operatives pair once).
- `stopQVACProvider()` to go dark. Idempotent.
- `heartbeat({ delegate: { providerPublicKey } })` to verify a peer is alive before delegating.

### Client (delegator) — mobile / another device
```ts
import { completion, heartbeat } from "@qvac/sdk";

const providerPublicKey = process.env.EXPO_PUBLIC_STATION_PEER_PUBKEY;
await heartbeat({ delegate: { providerPublicKey, timeout: 5000 } });

const run = completion({
  modelId,                              // model resolved on the provider
  history,
  // delegate to the station instead of running locally:
  delegate: { providerPublicKey },      // RPCOptions-style delegate target
});
for await (const t of run.tokenStream) { /* render */ }
```
> The exact `delegate` option shape lives in `RPCOptions`/`ProvideParams` in `@qvac/sdk` types — read the installed `.d.ts` and match it precisely. The pattern: provider calls `startQVACProvider`, consumers pass the provider's `publicKey`. `suspend()`/`resume()` stall and auto-recover delegated streams.

### Resolution policy (extend `apps/app/src/lib/inference/index.ts`)
```
ondevice  → run @qvac/sdk locally (Bare on mobile, Node on station)
delegate  → heartbeat(station); if alive, completion({delegate}); else fall back to ondevice
station   → call :11434 OpenAI-compat (browser on the station box)
```
Pick automatically: if on a constrained device and a paired station is reachable → `delegate`; else `ondevice`. Surface the chosen path in the UI (the "performance / load-distribution" story for judges).

## 2. Dead-drop messaging (operative ↔ handler)

Send an `IntelBrief` (or raw record) to a paired peer with no central server. Use **`hyperswarm`** directly (Holepunch; same DHT QVAC uses) for a simple encrypted drop.

### Dependencies
```
bun add hyperswarm b4a compact-encoding   # + sodium-native is pulled in transitively
```

### Pairing
- A **drop topic** = a 32-byte key shared out-of-band (QR / passphrase → hash to 32 bytes).
- Both peers `swarm.join(topic, { server, client })`; on `connection`, they get an end-to-end-encrypted Noise stream (Hyperswarm encrypts connections by default).

### Minimal API (`apps/app/src/lib/p2p/deaddrop.ts`)
```ts
export async function openDrop(topicKey: Buffer): Promise<DropChannel>;   // join swarm
export interface DropChannel {
  send(payload: Uint8Array): Promise<void>;   // a brief/record, app-layer-encrypted (AES-GCM) before send
  onMessage(cb: (payload: Uint8Array) => void): void;
  close(): Promise<void>;
}
```
- **Defense in depth:** even though Hyperswarm connections are encrypted, also AES-GCM the payload with a shared drop secret so a wrong-topic peer learns nothing. Reuse the dossier crypto from [03](./03-data-and-rag.md).
- Payload = `{ kind: "brief"|"record", ciphertext, iv, ts }`. Receiver decrypts → imports into its own dossier.
- This is **server-only / Bare** (raw sockets). On the station: a Node Route Handler or a small long-running worker. On mobile: Bare. Not in the browser bundle.

### Why two layers (Hyperswarm + delegated inference)
- Delegated inference = *compute* sharing (phone offloads to laptop).
- Dead-drop = *data* sharing (operative sends finished intel to handler).
Both reinforce the "edge/P2P AI, no central infra" Innovation + Performance scoring.

## 3. Security notes

- Never log peer public keys or topic keys to `artifacts/` in clear.
- Rotate the drop topic per mission (derive from mission passphrase).
- `QVAC_HYPERSWARM_SEED` is sensitive (it *is* the station identity) — keep it out of git; load from secure storage.
- Treat any inbound dropped payload as untrusted: validate schema, never auto-execute side-effecting tools from it.

## 4. Acceptance criteria

- [ ] Station prints a stable `publicKey`; restarting with the same `QVAC_HYPERSWARM_SEED` yields the same key.
- [ ] A second device delegates a `completion` to the station and renders streamed tokens (capture logs showing it ran on the provider, e.g. `getLoadedModelInfo` delegated=true).
- [ ] Falls back to on-device cleanly when the station is unreachable (`heartbeat` fails).
- [ ] Two app instances join a drop topic and exchange an encrypted `IntelBrief`; receiver imports it into its dossier.
- [ ] Wrong-topic / wrong-secret peer cannot read the payload.
- [ ] No data crosses the network except via these two explicit P2P paths.
